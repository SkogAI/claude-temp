     5â†’# Claude Code settings
     6â†’
     7â†’> Configure Claude Code with global and project-level settings, and environment variables.
     8â†’
     9â†’Claude Code offers a variety of settings to configure its behavior to meet your needs. You can configure Claude Code by running the `/config` command when using the interactive REPL, which opens a tabbed Settings interface where you can view status information and modify configuration options.
    10â†’
    11â†’## Configuration scopes
    12â†’
    13â†’Claude Code uses a **scope system** to determine where configurations apply and who they're shared with. Understanding scopes helps you decide how to configure Claude Code for personal use, team collaboration, or enterprise deployment.
    14â†’
    15â†’### Available scopes
    16â†’
    17â†’| Scope       | Location                                                                           | Who it affects                       | Shared with team?      |
    18â†’| :---------- | :--------------------------------------------------------------------------------- | :----------------------------------- | :--------------------- |
    19â†’| **Managed** | Server-managed settings, plist / registry, or system-level `managed-settings.json` | All users on the machine             | Yes (deployed by IT)   |
    20â†’| **User**    | `~/.claude/` directory                                                             | You, across all projects             | No                     |
    21â†’| **Project** | `.claude/` in repository                                                           | All collaborators on this repository | Yes (committed to git) |
    22â†’| **Local**   | `.claude/*.local.*` files                                                          | You, in this repository only         | No (gitignored)        |
    23â†’
    24â†’### When to use each scope
    25â†’
    26â†’**Managed scope** is for:
    27â†’
    28â†’* Security policies that must be enforced organization-wide
    29â†’* Compliance requirements that can't be overridden
    30â†’* Standardized configurations deployed by IT/DevOps
    31â†’
    32â†’**User scope** is best for:
    33â†’
    34â†’* Personal preferences you want everywhere (themes, editor settings)
    35â†’* Tools and plugins you use across all projects
    36â†’* API keys and authentication (stored securely)
    37â†’
    38â†’**Project scope** is best for:
    39â†’
    40â†’* Team-shared settings (permissions, hooks, MCP servers)
    41â†’* Plugins the whole team should have
    42â†’* Standardizing tooling across collaborators
    43â†’
    44â†’**Local scope** is best for:
    45â†’
    46â†’* Personal overrides for a specific project
    47â†’* Testing configurations before sharing with the team
    48â†’* Machine-specific settings that won't work for others
    49â†’
    50â†’### How scopes interact
    51â†’
    52â†’When the same setting is configured in multiple scopes, more specific scopes take precedence:
    53â†’
    54â†’1. **Managed** (highest) - can't be overridden by anything
    55â†’2. **Command line arguments** - temporary session overrides
    56â†’3. **Local** - overrides project and user settings
    57â†’4. **Project** - overrides user settings
    58â†’5. **User** (lowest) - applies when nothing else specifies the setting
    59â†’
    60â†’For example, if a permission is allowed in user settings but denied in project settings, the project setting takes precedence and the permission is blocked.
    61â†’
    62â†’### What uses scopes
    63â†’
    64â†’Scopes apply to many Claude Code features:
    65â†’
    66â†’| Feature         | User location             | Project location                   | Local location                 |
    67â†’| :-------------- | :------------------------ | :--------------------------------- | :----------------------------- |
    68â†’| **Settings**    | `~/.claude/settings.json` | `.claude/settings.json`            | `.claude/settings.local.json`  |
    69â†’| **Subagents**   | `~/.claude/agents/`       | `.claude/agents/`                  | â€”                              |
    70â†’| **MCP servers** | `~/.claude.json`          | `.mcp.json`                        | `~/.claude.json` (per-project) |
    71â†’| **Plugins**     | `~/.claude/settings.json` | `.claude/settings.json`            | `.claude/settings.local.json`  |
    72â†’| **CLAUDE.md**   | `~/.claude/CLAUDE.md`     | `CLAUDE.md` or `.claude/CLAUDE.md` | `CLAUDE.local.md`              |

    76â†’## Settings files
    77â†’
    78â†’The `settings.json` file is our official mechanism for configuring Claude
    79â†’Code through hierarchical settings:
    80â†’
    81â†’* **User settings** are defined in `~/.claude/settings.json` and apply to all
    82â†’  projects.
    83â†’* **Project settings** are saved in your project directory:
    84â†’  * `.claude/settings.json` for settings that are checked into source control and shared with your team
    85â†’  * `.claude/settings.local.json` for settings that are not checked in, useful for personal preferences and experimentation. Claude Code will configure git to ignore `.claude/settings.local.json` when it is created.
    86â†’* **Managed settings**: For organizations that need centralized control, Claude Code supports multiple delivery mechanisms for managed settings. All use the same JSON format and cannot be overridden by user or project settings:
    87â†’
    88â†’  * **Server-managed settings**: delivered from Anthropic's servers via the Claude.ai admin console. See [server-managed settings](/en/server-managed-settings).
    89â†’  * **MDM/OS-level policies**: delivered through native device management on macOS and Windows:
    90â†’    * macOS: `com.anthropic.claudecode` managed preferences domain (deployed via configuration profiles in Jamf, Kandji, or other MDM tools)
    91â†’    * Windows: `HKLM\SOFTWARE\Policies\ClaudeCode` registry key with a `Settings` value (REG\_SZ or REG\_EXPAND\_SZ) containing JSON (deployed via Group Policy or Intune)
    92â†’    * Windows (user-level): `HKCU\SOFTWARE\Policies\ClaudeCode` (lowest policy priority, only used when no admin-level source exists)
    93â†’  * **File-based**: `managed-settings.json` and `managed-mcp.json` deployed to system directories:
    94â†’    * macOS: `/Library/Application Support/ClaudeCode/`
    95â†’    * Linux and WSL: `/etc/claude-code/`
    96â†’    * Windows: `C:\Program Files\ClaudeCode\`
    97â†’
    98â†’  See [managed settings](/en/permissions#managed-only-settings) and [Managed MCP configuration](/en/mcp#managed-mcp-configuration) for details.
    99â†’
   100â†’  <Note>
   101â†’    Managed deployments can also restrict **plugin marketplace additions** using
   102â†’    `strictKnownMarketplaces`. For more information, see [Managed marketplace restrictions](/en/plugin-marketplaces#managed-marketplace-restrictions).
   103â†’  </Note>
   104â†’* **Other configuration** is stored in `~/.claude.json`. This file contains your preferences (theme, notification settings, editor mode), OAuth session, [MCP server](/en/mcp) configurations for user and local scopes, per-project state (allowed tools, trust settings), and various caches. Project-scoped MCP servers are stored separately in `.mcp.json`.
   105â†’
   106â†’<Note>
   107â†’  Claude Code automatically creates timestamped backups of configuration files and retains the five most recent backups to prevent data loss.
   108â†’</Note>
   109â†’
   110â†’```JSON Example settings.json theme={null}
   111â†’{
   112â†’  "$schema": "https://json.schemastore.org/claude-code-settings.json",
   113â†’  "permissions": {
   114â†’    "allow": [
   115â†’      "Bash(npm run lint)",
   116â†’      "Bash(npm run test *)",
   117â†’      "Read(~/.zshrc)"
   118â†’    ],
   119â†’    "deny": [
   120â†’      "Bash(curl *)",
   121â†’      "Read(./.env)",
   122â†’      "Read(./.env.*)",
   123â†’      "Read(./secrets/**)"
   124â†’    ]
   125â†’  },
   126â†’  "env": {
   127â†’    "CLAUDE_CODE_ENABLE_TELEMETRY": "1",
   128â†’    "OTEL_METRICS_EXPORTER": "otlp"
   129â†’  },
   130â†’  "companyAnnouncements": [
   131â†’    "Welcome to Acme Corp! Review our code guidelines at docs.acme.com",
   132â†’    "Reminder: Code reviews required for all PRs",
   133â†’    "New security policy in effect"
   134â†’  ]
   135â†’}
   136â†’```
   137â†’
   138â†’The `$schema` line in the example above points to the [official JSON schema](https://json.schemastore.org/claude-code-settings.json) for Claude Code settings. Adding it to your `settings.json` enables autocomplete and inline validation in VS Code, Cursor, and any other editor that supports JSON schema validation.

   202â†’### Permission rule syntax
   203â†’
   204â†’Permission rules follow the format `Tool` or `Tool(specifier)`. Rules are evaluated in order: deny rules first, then ask, then allow. The first matching rule wins.
   205â†’
   206â†’Quick examples:
   207â†’
   208â†’| Rule                           | Effect                                   |
   209â†’| :----------------------------- | :--------------------------------------- |
   210â†’| `Bash`                         | Matches all Bash commands                |
   211â†’| `Bash(npm run *)`              | Matches commands starting with `npm run` |
   212â†’| `Read(./.env)`                 | Matches reading the `.env` file          |
   213â†’| `WebFetch(domain:example.com)` | Matches fetch requests to example.com    |
   214â†’
   215â†’For the complete rule syntax reference, including wildcard behavior, tool-specific patterns for Read, Edit, WebFetch, MCP, and Agent rules, and security limitations of Bash patterns, see [Permission rule syntax](/en/permissions#permission-rule-syntax).

   242â†’
   243â†’| Prefix            | Meaning                                     | Example                                |
   244â†’| :---------------- | :------------------------------------------ | :------------------------------------- |
   245â†’| `//`              | Absolute path from filesystem root          | `//tmp/build` becomes `/tmp/build`     |
   246â†’| `~/`              | Relative to home directory                  | `~/.kube` becomes `$HOME/.kube`        |
   247â†’| `/`               | Relative to the settings file's directory   | `/build` becomes `$SETTINGS_DIR/build` |
   248â†’| `./` or no prefix | Relative path (resolved by sandbox runtime) | `./output`                             |
   249â†’
   250â†’**Configuration example:**
   251â†’
   252â†’```json  theme={null}
   253â†’{
   254â†’  "sandbox": {
   255â†’    "enabled": true,
   256â†’    "autoAllowBashIfSandboxed": true,
   257â†’    "excludedCommands": ["docker"],
   258â†’    "filesystem": {
   259â†’      "allowWrite": ["//tmp/build", "~/.kube"],
   260â†’      "denyRead": ["~/.aws/credentials"]
   261â†’    },
   262â†’    "network": {
   263â†’      "allowedDomains": ["github.com", "*.npmjs.org", "registry.yarnpkg.com"],
   264â†’      "allowUnixSockets": [
   265â†’        "/var/run/docker.sock"
   266â†’      ],
   267â†’      "allowLocalBinding": true
   268â†’    }
   269â†’  }
   270â†’}
   271â†’```
   272â†’
   273â†’**Filesystem and network restrictions** can be configured in two ways that are merged together:
   274â†’
   275â†’* **`sandbox.filesystem` settings** (shown above): Control paths at the OS-level sandbox boundary. These restrictions apply to all subprocess commands (e.g., `kubectl`, `terraform`, `npm`), not just Claude's file tools.
   276â†’* **Permission rules**: Use `Edit` allow/deny rules to control Claude's file tool access, `Read` deny rules to block reads, and `WebFetch` allow/deny rules to control network domains. Paths from these rules are also merged into the sandbox configuration.
   277â†’

   304â†’**Example:**
   305â†’
   306â†’```json  theme={null}

   354â†’### Hook configuration
   355â†’
   356â†’These settings control which hooks are allowed to run and what HTTP hooks can access. The `allowManagedHooksOnly` setting can only be configured in [managed settings](#settings-files). The URL and env var allowlists can be set at any settings level and merge across sources.
   357â†’
   358â†’**Behavior when `allowManagedHooksOnly` is `true`:**
   359â†’
   360â†’* Managed hooks and SDK hooks are loaded
   361â†’* User hooks, project hooks, and plugin hooks are blocked
   362â†’
   363â†’**Restrict HTTP hook URLs:**
   364â†’
   365â†’Limit which URLs HTTP hooks can target. Supports `*` as a wildcard for matching. When the array is defined, HTTP hooks targeting non-matching URLs are silently blocked.
   366â†’
   367â†’```json  theme={null}
   368â†’{
   369â†’  "allowedHttpHookUrls": ["https://hooks.example.com/*", "http://localhost:*"]
   370â†’}
   371â†’```
   372â†’
   373â†’**Restrict HTTP hook environment variables:**
   374â†’
   375â†’Limit which environment variable names HTTP hooks can interpolate into header values. Each hook's effective `allowedEnvVars` is the intersection of its own list and this setting.
   376â†’
   377â†’```json  theme={null}
   378â†’{
   379â†’  "httpHookAllowedEnvVars": ["MY_TOKEN", "HOOK_SECRET"]
   380â†’}
   381â†’```

   383â†’### Settings precedence
   384â†’
   385â†’Settings apply in order of precedence. From highest to lowest:
   386â†’
   387â†’1. **Managed settings** ([server-managed](/en/server-managed-settings), [MDM/OS-level policies](#configuration-scopes), or [managed settings](/en/settings#settings-files))
   388â†’   * Policies deployed by IT through server delivery, MDM configuration profiles, registry policies, or managed settings files
   389â†’   * Cannot be overridden by user or project settings
   390â†’   * Within the managed tier, precedence is: server-managed > MDM/OS-level policies > `managed-settings.json` > HKCU registry (Windows only). Only one managed source is used; sources do not merge.
   391â†’
   392â†’2. **Command line arguments**
   393â†’   * Temporary overrides for a specific session
   394â†’
   395â†’3. **Local project settings** (`.claude/settings.local.json`)
   396â†’   * Personal project-specific settings
   397â†’
   398â†’4. **Shared project settings** (`.claude/settings.json`)
   399â†’   * Team-shared project settings in source control
   400â†’
   401â†’5. **User settings** (`~/.claude/settings.json`)
   402â†’   * Personal global settings
   403â†’
   404â†’This hierarchy ensures that organizational policies are always enforced while still allowing teams and individuals to customize their experience.
   405â†’
   406â†’For example, if your user settings allow `Bash(npm run *)` but a project's shared settings deny it, the project setting takes precedence and the command is blocked.
   407â†’
   408â†’<Note>
   409â†’  **Array settings merge across scopes.** When the same array-valued setting (such as `sandbox.filesystem.allowWrite` or `permissions.allow`) appears in multiple scopes, the arrays are **concatenated and deduplicated**, not replaced. This means lower-priority scopes can add entries without overriding those set by higher-priority scopes, and vice versa. For example, if managed settings set `allowWrite` to `["//opt/company-tools"]` and a user adds `["~/.kube"]`, both paths are included in the final configuration.
   410â†’</Note>
   411â†’
   412â†’### Verify active settings
   413â†’
   414â†’Run `/status` inside Claude Code to see which settings sources are active and where they come from. The output shows each configuration layer (managed, user, project) along with its origin, such as `Enterprise managed settings (remote)`, `Enterprise managed settings (plist)`, `Enterprise managed settings (HKLM)`, or `Enterprise managed settings (file)`. If a settings file contains errors, `/status` reports the issue so you can fix it.
   415â†’
   416â†’### Key points about the configuration system
   417â†’
   418â†’* **Memory files (`CLAUDE.md`)**: Contain instructions and context that Claude loads at startup
   419â†’* **Settings files (JSON)**: Configure permissions, environment variables, and tool behavior
   420â†’* **Skills**: Custom prompts that can be invoked with `/skill-name` or loaded by Claude automatically
   421â†’* **MCP servers**: Extend Claude Code with additional tools and integrations
   422â†’* **Precedence**: Higher-level configurations (Managed) override lower-level ones (User/Project)
   423â†’* **Inheritance**: Settings are merged, with more specific settings adding to or overriding broader ones
   424â†’
   425â†’### System prompt
   426â†’
   427â†’Claude Code's internal system prompt is not published. To add custom instructions, use `CLAUDE.md` files or the `--append-system-prompt` flag.
   428â†’
   429â†’### Excluding sensitive files
   430â†’
   431â†’To prevent Claude Code from accessing files containing sensitive information like API keys, secrets, and environment files, use the `permissions.deny` setting in your `.claude/settings.json` file:
   432â†’
   433â†’```json  theme={null}
   434â†’{
   435â†’  "permissions": {
   436â†’    "deny": [
   437â†’      "Read(./.env)",
   438â†’      "Read(./.env.*)",
   439â†’      "Read(./secrets/**)",
   440â†’      "Read(./config/credentials.json)",
   441â†’      "Read(./build)"
   442â†’    ]
   443â†’  }
   444â†’}
   445â†’```
   446â†’
   447â†’This replaces the deprecated `ignorePatterns` configuration. Files matching these patterns are excluded from file discovery and search results, and read operations on these files are denied.

   458â†’## Plugin configuration
   459â†’
   460â†’Claude Code supports a plugin system that lets you extend functionality with skills, agents, hooks, and MCP servers. Plugins are distributed through marketplaces and can be configured at both user and repository levels.
   461â†’
   462â†’### Plugin settings
   463â†’
   464â†’Plugin-related settings in `settings.json`:
   465â†’
   466â†’```json  theme={null}
   467â†’{
   468â†’  "enabledPlugins": {
   469â†’    "formatter@acme-tools": true,
   470â†’    "deployer@acme-tools": true,
   471â†’    "analyzer@security-plugins": false
   472â†’  },
   473â†’  "extraKnownMarketplaces": {
   474â†’    "acme-tools": {
   475â†’      "source": "github",
   476â†’      "repo": "acme-corp/claude-plugins"
   477â†’    }
   478â†’  }
   479â†’}
   480â†’```
   481â†’
   482â†’#### `enabledPlugins`
   483â†’
   484â†’Controls which plugins are enabled. Format: `"plugin-name@marketplace-name": true/false`
   485â†’
   486â†’**Scopes**:
   487â†’
   488â†’* **User settings** (`~/.claude/settings.json`): Personal plugin preferences
   489â†’* **Project settings** (`.claude/settings.json`): Project-specific plugins shared with team
   490â†’* **Local settings** (`.claude/settings.local.json`): Per-machine overrides (not committed)
   491â†’
   492â†’**Example**:
   493â†’
   494â†’```json  theme={null}
   495â†’{
   496â†’  "enabledPlugins": {
   497â†’    "code-formatter@team-tools": true,
   498â†’    "deployment-tools@team-tools": true,
   499â†’    "experimental-features@personal": false
   500â†’  }
   501â†’}
   502â†’```
   503â†’
   504â†’#### `extraKnownMarketplaces`
   505â†’
   506â†’Defines additional marketplaces that should be made available for the repository. Typically used in repository-level settings to ensure team members have access to required plugin sources.
   507â†’
   508â†’**When a repository includes `extraKnownMarketplaces`**:
   509â†’
   510â†’1. Team members are prompted to install the marketplace when they trust the folder
   511â†’2. Team members are then prompted to install plugins from that marketplace
   512â†’3. Users can skip unwanted marketplaces or plugins (stored in user settings)
   513â†’4. Installation respects trust boundaries and requires explicit consent
   514â†’
   515â†’**Example**:
   516â†’
   517â†’```json  theme={null}
   518â†’{
   519â†’  "extraKnownMarketplaces": {
   520â†’    "acme-tools": {
   521â†’      "source": {
   522â†’        "source": "github",
   523â†’        "repo": "acme-corp/claude-plugins"
   524â†’      }
   525â†’    },
   526â†’    "security-plugins": {
   527â†’      "source": {
   528â†’        "source": "git",
   529â†’        "url": "https://git.example.com/security/plugins.git"
   530â†’      }
   531â†’    }
   532â†’  }
   533â†’}
   534â†’```
   535â†’
   536â†’**Marketplace source types**:
   537â†’
   538â†’* `github`: GitHub repository (uses `repo`)
   539â†’* `git`: Any git URL (uses `url`)
   540â†’* `directory`: Local filesystem path (uses `path`, for development only)
   541â†’* `hostPattern`: regex pattern to match marketplace hosts (uses `hostPattern`)
   542â†’
   543â†’#### `strictKnownMarketplaces`
   544â†’
   545â†’**Managed settings only**: Controls which plugin marketplaces users are allowed to add. This setting can only be configured in [managed settings](/en/settings#settings-files) and provides administrators with strict control over marketplace sources.
   546â†’
   547â†’**Managed settings file locations**:
   548â†’
   549â†’* **macOS**: `/Library/Application Support/ClaudeCode/managed-settings.json`
   550â†’* **Linux and WSL**: `/etc/claude-code/managed-settings.json`
   551â†’* **Windows**: `C:\Program Files\ClaudeCode\managed-settings.json`
   552â†’
   553â†’**Key characteristics**:
   554â†’
   555â†’* Only available in managed settings (`managed-settings.json`)
   556â†’* Cannot be overridden by user or project settings (highest precedence)
   557â†’* Enforced BEFORE network/filesystem operations (blocked sources never execute)
   558â†’* Uses exact matching for source specifications (including `ref`, `path` for git sources), except `hostPattern`, which uses regex matching
   559â†’
   560â†’**Allowlist behavior**:
   561â†’
   562â†’* `undefined` (default): No restrictions - users can add any marketplace
   563â†’* Empty array `[]`: Complete lockdown - users cannot add any new marketplaces
   564â†’* List of sources: Users can only add marketplaces that match exactly
   565â†’
   566â†’**All supported source types**:
   567â†’
   568â†’The allowlist supports seven marketplace source types. Most sources use exact matching, while `hostPattern` uses regex matching against the marketplace host.
   569â†’
   570â†’1. **GitHub repositories**:
   571â†’
   572â†’```json  theme={null}
   573â†’{ "source": "github", "repo": "acme-corp/approved-plugins" }
   574â†’{ "source": "github", "repo": "acme-corp/security-tools", "ref": "v2.0" }
   575â†’{ "source": "github", "repo": "acme-corp/plugins", "ref": "main", "path": "marketplace" }
   576â†’```
   577â†’
   578â†’Fields: `repo` (required), `ref` (optional: branch/tag/SHA), `path` (optional: subdirectory)
   579â†’
   580â†’2. **Git repositories**:
   581â†’
   582â†’```json  theme={null}
   583â†’{ "source": "git", "url": "https://gitlab.example.com/tools/plugins.git" }
   584â†’{ "source": "git", "url": "https://bitbucket.org/acme-corp/plugins.git", "ref": "production" }
   585â†’{ "source": "git", "url": "ssh://git@git.example.com/plugins.git", "ref": "v3.1", "path": "approved" }
   586â†’```
   587â†’
   588â†’Fields: `url` (required), `ref` (optional: branch/tag/SHA), `path` (optional: subdirectory)
   589â†’
   590â†’3. **URL-based marketplaces**:
   591â†’
   592â†’```json  theme={null}
   593â†’{ "source": "url", "url": "https://plugins.example.com/marketplace.json" }
   594â†’{ "source": "url", "url": "https://cdn.example.com/marketplace.json", "headers": { "Authorization": "Bearer ${TOKEN}" } }
   595â†’```
   596â†’
   597â†’Fields: `url` (required), `headers` (optional: HTTP headers for authenticated access)
   598â†’
   599â†’<Note>
   600â†’  URL-based marketplaces only download the `marketplace.json` file. They do not download plugin files from the server. Plugins in URL-based marketplaces must use external sources (GitHub, npm, or git URLs) rather than relative paths. For plugins with relative paths, use a Git-based marketplace instead. See [Troubleshooting](/en/plugin-marketplaces#plugins-with-relative-paths-fail-in-url-based-marketplaces) for details.
   601â†’</Note>
   602â†’
   603â†’4. **NPM packages**:
   604â†’
   605â†’```json  theme={null}
   606â†’{ "source": "npm", "package": "@acme-corp/claude-plugins" }
   607â†’{ "source": "npm", "package": "@acme-corp/approved-marketplace" }
   608â†’```
   609â†’
   610â†’Fields: `package` (required, supports scoped packages)
   611â†’
   612â†’5. **File paths**:
   613â†’
   614â†’```json  theme={null}
   615â†’{ "source": "file", "path": "/usr/local/share/claude/acme-marketplace.json" }
   616â†’{ "source": "file", "path": "/opt/acme-corp/plugins/marketplace.json" }
   617â†’```
   618â†’
   619â†’Fields: `path` (required: absolute path to marketplace.json file)
   620â†’
   621â†’6. **Directory paths**:
   622â†’
   623â†’```json  theme={null}
   624â†’{ "source": "directory", "path": "/usr/local/share/claude/acme-plugins" }
   625â†’{ "source": "directory", "path": "/opt/acme-corp/approved-marketplaces" }
   626â†’```
   627â†’
   628â†’Fields: `path` (required: absolute path to directory containing `.claude-plugin/marketplace.json`)
   629â†’
   630â†’7. **Host pattern matching**:
   631â†’
   632â†’```json  theme={null}
   633â†’{ "source": "hostPattern", "hostPattern": "^github\\.example\\.com$" }
   634â†’{ "source": "hostPattern", "hostPattern": "^gitlab\\.internal\\.example\\.com$" }
   635â†’```
   636â†’
   637â†’Fields: `hostPattern` (required: regex pattern to match against the marketplace host)
   638â†’
   639â†’Use host pattern matching when you want to allow all marketplaces from a specific host without enumerating each repository individually. This is useful for organizations with internal GitHub Enterprise or GitLab servers where developers create their own marketplaces.
   640â†’
   641â†’Host extraction by source type:
   642â†’
   643â†’* `github`: always matches against `github.com`
   644â†’* `git`: extracts hostname from the URL (supports both HTTPS and SSH formats)
   645â†’* `url`: extracts hostname from the URL
   646â†’* `npm`, `file`, `directory`: not supported for host pattern matching
   647â†’
   648â†’**Configuration examples**:
   649â†’
   650â†’Example: allow specific marketplaces only:
   651â†’
   652â†’```json  theme={null}
   653â†’{
   654â†’  "strictKnownMarketplaces": [
   655â†’    {
   656â†’      "source": "github",
   657â†’      "repo": "acme-corp/approved-plugins"
   658â†’    },
   659â†’    {
   660â†’      "source": "github",
   661â†’      "repo": "acme-corp/security-tools",
   662â†’      "ref": "v2.0"
   663â†’    },
   664â†’    {
   665â†’      "source": "url",
   666â†’      "url": "https://plugins.example.com/marketplace.json"
   667â†’    },
   668â†’    {
   669â†’      "source": "npm",
   670â†’      "package": "@acme-corp/compliance-plugins"
   671â†’    }
   672â†’  ]
   673â†’}
   674â†’```
   675â†’
   676â†’Example - Disable all marketplace additions:
   677â†’
   678â†’```json  theme={null}
   679â†’{
   680â†’  "strictKnownMarketplaces": []
   681â†’}
   682â†’```
   683â†’
   684â†’Example: allow all marketplaces from an internal git server:
   685â†’
   686â†’```json  theme={null}
   687â†’{
   688â†’  "strictKnownMarketplaces": [
   689â†’    {
   690â†’      "source": "hostPattern",
   691â†’      "hostPattern": "^github\\.example\\.com$"
   692â†’    }
   693â†’  ]
   694â†’}
   695â†’```
   696â†’
   697â†’**Exact matching requirements**:
   698â†’
   699â†’Marketplace sources must match **exactly** for a user's addition to be allowed. For git-based sources (`github` and `git`), this includes all optional fields:
   700â†’
   701â†’* The `repo` or `url` must match exactly
   702â†’* The `ref` field must match exactly (or both be undefined)
   703â†’* The `path` field must match exactly (or both be undefined)
   704â†’
   705â†’Examples of sources that **do NOT match**:
   706â†’
   707â†’```json  theme={null}
   708â†’// These are DIFFERENT sources:
   709â†’{ "source": "github", "repo": "acme-corp/plugins" }
   710â†’{ "source": "github", "repo": "acme-corp/plugins", "ref": "main" }
   711â†’
   712â†’// These are also DIFFERENT:
   713â†’{ "source": "github", "repo": "acme-corp/plugins", "path": "marketplace" }
   714â†’{ "source": "github", "repo": "acme-corp/plugins" }
   715â†’```
   716â†’
   717â†’**Comparison with `extraKnownMarketplaces`**:
   718â†’
   719â†’| Aspect                | `strictKnownMarketplaces`            | `extraKnownMarketplaces`             |
   720â†’| --------------------- | ------------------------------------ | ------------------------------------ |
   721â†’| **Purpose**           | Organizational policy enforcement    | Team convenience                     |
   722â†’| **Settings file**     | `managed-settings.json` only         | Any settings file                    |
   723â†’| **Behavior**          | Blocks non-allowlisted additions     | Auto-installs missing marketplaces   |
   724â†’| **When enforced**     | Before network/filesystem operations | After user trust prompt              |
   725â†’| **Can be overridden** | No (highest precedence)              | Yes (by higher precedence settings)  |
   726â†’| **Source format**     | Direct source object                 | Named marketplace with nested source |
   727â†’| **Use case**          | Compliance, security restrictions    | Onboarding, standardization          |
   728â†’
   729â†’**Format difference**:
   730â†’
   731â†’`strictKnownMarketplaces` uses direct source objects:
   732â†’
   733â†’```json  theme={null}
   734â†’{
   735â†’  "strictKnownMarketplaces": [

   753â†’**Important notes**:
   754â†’
   755â†’* Restrictions are checked BEFORE any network requests or filesystem operations
   756â†’* When blocked, users see clear error messages indicating the source is blocked by managed policy
   757â†’* The restriction applies only to adding NEW marketplaces; previously installed marketplaces remain accessible
   758â†’* Managed settings have the highest precedence and cannot be overridden

   909â†’### Bash tool behavior
   910â†’
   911â†’The Bash tool executes shell commands with the following persistence behavior:
   912â†’
   913â†’* **Working directory persists**: When Claude changes the working directory (for example, `cd /path/to/dir`), subsequent Bash commands will execute in that directory. You can use `CLAUDE_BASH_MAINTAIN_PROJECT_WORKING_DIR=1` to reset to the project directory after each command.
   914â†’* **Environment variables do NOT persist**: Environment variables set in one Bash command (for example, `export MY_VAR=value`) are **not** available in subsequent Bash commands. Each Bash command runs in a fresh shell environment.
   915â†’
   916â†’To make environment variables available in Bash commands, you have **three options**:
   917â†’
   918â†’**Option 1: Activate environment before starting Claude Code** (simplest approach)
   919â†’
   920â†’Activate your virtual environment in your terminal before launching Claude Code:
   921â†’
   922â†’```bash  theme={null}
   923â†’conda activate myenv
   924â†’# or: source /path/to/venv/bin/activate
   925â†’claude
   926â†’```
   927â†’
   928â†’This works for shell environments but environment variables set within Claude's Bash commands will not persist between commands.
   929â†’
   930â†’**Option 2: Set CLAUDE\_ENV\_FILE before starting Claude Code** (persistent environment setup)
   931â†’
   932â†’Export the path to a shell script containing your environment setup:
   933â†’
   934â†’```bash  theme={null}
   935â†’export CLAUDE_ENV_FILE=/path/to/env-setup.sh
   936â†’claude
   937â†’```
   938â†’
   939â†’Where `/path/to/env-setup.sh` contains:
   940â†’
   941â†’```bash  theme={null}
   942â†’conda activate myenv
   943â†’# or: source /path/to/venv/bin/activate
   944â†’# or: export MY_VAR=value
   945â†’```
   946â†’
   947â†’Claude Code will source this file before each Bash command, making the environment persistent across all commands.
   948â†’
   949â†’**Option 3: Use a SessionStart hook** (project-specific configuration)
   950â†’
   951â†’Configure in `.claude/settings.json`:
   952â†’
   953â†’```json  theme={null}
   954â†’{
   955â†’  "hooks": {
   956â†’    "SessionStart": [{
   957â†’      "matcher": "startup",
   958â†’      "hooks": [{
   959â†’        "type": "command",
   960â†’        "command": "echo 'conda activate myenv' >> \"$CLAUDE_ENV_FILE\""
   961â†’      }]
   962â†’    }]
   963â†’  }
   964â†’}
   965â†’```
   966â†’
   967â†’The hook writes to `$CLAUDE_ENV_FILE`, which is then sourced before each Bash command. This is ideal for team-shared project configurations.
   968â†’
   969â†’See [SessionStart hooks](/en/hooks#persist-environment-variables) for more details on Option 3.
