    12‚Üí    12‚ÜíA simple, open format for giving agents new capabilities and expertise.
    13‚Üí    13‚Üí
    14‚Üí    14‚ÜíAgent Skills are folders of instructions, scripts, and resources that agents can discover and use to do things more accurately and efficiently.
    15‚Üí    15‚Üí‚Äã
    16‚Üí    16‚ÜíWhy Agent Skills?
    17‚Üí    17‚ÜíAgents are increasingly capable, but often don‚Äôt have the context they need to do real work reliably. Skills solve this by giving agents access to procedural knowledge and company-, team-, and user-specific context they can load on demand. Agents with access to a set of skills can extend their capabilities based on the task they‚Äôre working on.
    18‚Üí    18‚ÜíFor skill authors: Build capabilities once and deploy them across multiple agent products.
    19‚Üí    19‚ÜíFor compatible agents: Support for skills lets end users give agents new capabilities out of the box.
    20‚Üí    20‚ÜíFor teams and enterprises: Capture organizational knowledge in portable, version-controlled packages.
    21‚Üí    21‚Üí‚Äã
    22‚Üí    22‚ÜíWhat can Agent Skills enable?
    23‚Üí    23‚ÜíDomain expertise: Package specialized knowledge into reusable instructions, from legal review processes to data analysis pipelines.
    24‚Üí    24‚ÜíNew capabilities: Give agents new capabilities (e.g. creating presentations, building MCP servers, analyzing datasets).
    25‚Üí    25‚ÜíRepeatable workflows: Turn multi-step tasks into consistent and auditable workflows.
    26‚Üí    26‚ÜíInteroperability: Reuse the same skill across different skills-compatible agent products.

    44‚Üí    44‚ÜíWhat are skills?
    45‚Üí    45‚Üí
    46‚Üí    46‚ÜíCopy page
    47‚Üí    47‚Üí
    48‚Üí    48‚ÜíAgent Skills are a lightweight, open format for extending AI agent capabilities with specialized knowledge and workflows.
    49‚Üí    49‚Üí
    50‚Üí    50‚ÜíAt its core, a skill is a folder containing a SKILL.md file. This file includes metadata (name and description, at minimum) and instructions that tell an agent how to perform a specific task. Skills can also bundle scripts, templates, and reference materials.
    51‚Üí    51‚Üímy-skill/
    52‚Üí    52‚Üí‚îú‚îÄ‚îÄ SKILL.md          # Required: instructions + metadata
    53‚Üí    53‚Üí‚îú‚îÄ‚îÄ scripts/          # Optional: executable code
    54‚Üí    54‚Üí‚îú‚îÄ‚îÄ references/       # Optional: documentation
    55‚Üí    55‚Üí‚îî‚îÄ‚îÄ assets/           # Optional: templates, resources
    56‚Üí    56‚Üí‚Äã
    57‚Üí    57‚ÜíHow skills work
    58‚Üí    58‚ÜíSkills use progressive disclosure to manage context efficiently:
    59‚Üí    59‚ÜíDiscovery: At startup, agents load only the name and description of each available skill, just enough to know when it might be relevant.
    60‚Üí    60‚ÜíActivation: When a task matches a skill‚Äôs description, the agent reads the full SKILL.md instructions into context.
    61‚Üí    61‚ÜíExecution: The agent follows the instructions, optionally loading referenced files or executing bundled code as needed.
    62‚Üí    62‚ÜíThis approach keeps agents fast while giving them access to more context on demand.
    63‚Üí    63‚Üí‚Äã
    64‚Üí    64‚ÜíThe SKILL.md file
    65‚Üí    65‚ÜíEvery skill starts with a SKILL.md file containing YAML frontmatter and Markdown instructions:
    66‚Üí    66‚Üí---
    67‚Üí    67‚Üíname: pdf-processing
    68‚Üí    68‚Üídescription: Extract text and tables from PDF files, fill forms, merge documents.
    69‚Üí    69‚Üí---
    70‚Üí    70‚Üí
    71‚Üí    71‚Üí# PDF Processing
    72‚Üí    72‚Üí
    73‚Üí    73‚Üí## When to use this skill
    74‚Üí    74‚ÜíUse this skill when the user needs to work with PDF files...
    75‚Üí    75‚Üí
    76‚Üí    76‚Üí## How to extract text
    77‚Üí    77‚Üí1. Use pdfplumber for text extraction...
    78‚Üí    78‚Üí
    79‚Üí    79‚Üí## How to fill forms
    80‚Üí    80‚Üí...
    81‚Üí    81‚ÜíThe following frontmatter is required at the top of SKILL.md:
    82‚Üí    82‚Üíname: A short identifier
    83‚Üí    83‚Üídescription: When to use this skill
    84‚Üí    84‚ÜíThe Markdown body contains the actual instructions and has no specific restrictions on structure or content.
    85‚Üí    85‚ÜíThis simple format has some key advantages:
    86‚Üí    86‚ÜíSelf-documenting: A skill author or user can read a SKILL.md and understand what it does, making skills easy to audit and improve.
    87‚Üí    87‚ÜíExtensible: Skills can range in complexity from just text instructions to executable code, assets, and templates.
    88‚Üí    88‚ÜíPortable: Skills are just files, so they‚Äôre easy to edit, version, and share.

   101‚Üí   101‚ÜíThe complete format specification for Agent Skills.
   102‚Üí   102‚Üí
   103‚Üí   103‚ÜíThis document defines the Agent Skills format.
   104‚Üí   104‚Üí‚Äã
   105‚Üí   105‚ÜíDirectory structure
   106‚Üí   106‚ÜíA skill is a directory containing at minimum a SKILL.md file:
   107‚Üí   107‚Üískill-name/
   108‚Üí   108‚Üí‚îî‚îÄ‚îÄ SKILL.md          # Required
   109‚Üí   109‚ÜíYou can optionally include additional directories such as scripts/, references/, and assets/ to support your skill.
   110‚Üí   110‚Üí‚Äã
   111‚Üí   111‚ÜíSKILL.md format
   112‚Üí   112‚ÜíThe SKILL.md file must contain YAML frontmatter followed by Markdown content.
   113‚Üí   113‚Üí‚Äã
   114‚Üí   114‚ÜíFrontmatter (required)
   115‚Üí   115‚Üí---
   116‚Üí   116‚Üíname: skill-name
   117‚Üí   117‚Üídescription: A description of what this skill does and when to use it.
   118‚Üí   118‚Üí---
   119‚Üí   119‚ÜíWith optional fields:
   120‚Üí   120‚Üí---
   121‚Üí   121‚Üíname: pdf-processing
   122‚Üí   122‚Üídescription: Extract text and tables from PDF files, fill forms, merge documents.
   123‚Üí   123‚Üílicense: Apache-2.0
   124‚Üí   124‚Üímetadata:
   125‚Üí   125‚Üí  author: example-org
   126‚Üí   126‚Üí  version: "1.0"
   127‚Üí   127‚Üí---
   128‚Üí   128‚ÜíField	Required	Constraints
   129‚Üí   129‚Üíname	Yes	Max 64 characters. Lowercase letters, numbers, and hyphens only. Must not start or end with a hyphen.
   130‚Üí   130‚Üídescription	Yes	Max 1024 characters. Non-empty. Describes what the skill does and when to use it.
   131‚Üí   131‚Üílicense	No	License name or reference to a bundled license file.
   132‚Üí   132‚Üícompatibility	No	Max 500 characters. Indicates environment requirements (intended product, system packages, network access, etc.).
   133‚Üí   133‚Üímetadata	No	Arbitrary key-value mapping for additional metadata.
   134‚Üí   134‚Üíallowed-tools	No	Space-delimited list of pre-approved tools the skill may use. (Experimental)
   135‚Üí   135‚Üí‚Äã
   136‚Üí   136‚Üíname field
   137‚Üí   137‚ÜíThe required name field:
   138‚Üí   138‚ÜíMust be 1-64 characters
   139‚Üí   139‚ÜíMay only contain unicode lowercase alphanumeric characters and hyphens (a-z and -)
   140‚Üí   140‚ÜíMust not start or end with -
   141‚Üí   141‚ÜíMust not contain consecutive hyphens (--)
   142‚Üí   142‚ÜíMust match the parent directory name
   143‚Üí   143‚ÜíValid examples:
   144‚Üí   144‚Üíname: pdf-processing
   145‚Üí   145‚Üíname: data-analysis
   146‚Üí   146‚Üíname: code-review
   147‚Üí   147‚ÜíInvalid examples:
   148‚Üí   148‚Üíname: PDF-Processing  # uppercase not allowed
   149‚Üí   149‚Üíname: -pdf  # cannot start with hyphen
   150‚Üí   150‚Üíname: pdf--processing  # consecutive hyphens not allowed
   151‚Üí   151‚Üí‚Äã
   152‚Üí   152‚Üídescription field
   153‚Üí   153‚ÜíThe required description field:
   154‚Üí   154‚ÜíMust be 1-1024 characters
   155‚Üí   155‚ÜíShould describe both what the skill does and when to use it
   156‚Üí   156‚ÜíShould include specific keywords that help agents identify relevant tasks
   157‚Üí   157‚ÜíGood example:
   158‚Üí   158‚Üídescription: Extracts text and tables from PDF files, fills PDF forms, and merges multiple PDFs. Use when working with PDF documents or when the user mentions PDFs, forms, or document extraction.
   159‚Üí   159‚ÜíPoor example:
   160‚Üí   160‚Üídescription: Helps with PDFs.
   161‚Üí   161‚Üí‚Äã
   162‚Üí   162‚Üílicense field
   163‚Üí   163‚ÜíThe optional license field:
   164‚Üí   164‚ÜíSpecifies the license applied to the skill
   165‚Üí   165‚ÜíWe recommend keeping it short (either the name of a license or the name of a bundled license file)
   166‚Üí   166‚ÜíExample:
   167‚Üí   167‚Üílicense: Proprietary. LICENSE.txt has complete terms
   168‚Üí   168‚Üí‚Äã
   169‚Üí   169‚Üícompatibility field
   170‚Üí   170‚ÜíThe optional compatibility field:
   171‚Üí   171‚ÜíMust be 1-500 characters if provided
   172‚Üí   172‚ÜíShould only be included if your skill has specific environment requirements
   173‚Üí   173‚ÜíCan indicate intended product, required system packages, network access needs, etc.
   174‚Üí   174‚ÜíExamples:
   175‚Üí   175‚Üícompatibility: Designed for Claude Code (or similar products)
   176‚Üí   176‚Üícompatibility: Requires git, docker, jq, and access to the internet
   177‚Üí   177‚ÜíMost skills do not need the compatibility field.
   178‚Üí   178‚Üí‚Äã
   179‚Üí   179‚Üímetadata field
   180‚Üí   180‚ÜíThe optional metadata field:
   181‚Üí   181‚ÜíA map from string keys to string values
   182‚Üí   182‚ÜíClients can use this to store additional properties not defined by the Agent Skills spec
   183‚Üí   183‚ÜíWe recommend making your key names reasonably unique to avoid accidental conflicts
   184‚Üí   184‚ÜíExample:
   185‚Üí   185‚Üímetadata:
   186‚Üí   186‚Üí  author: example-org
   187‚Üí   187‚Üí  version: "1.0"
   188‚Üí   188‚Üí‚Äã
   189‚Üí   189‚Üíallowed-tools field
   190‚Üí   190‚ÜíThe optional allowed-tools field:
   191‚Üí   191‚ÜíA space-delimited list of tools that are pre-approved to run
   192‚Üí   192‚ÜíExperimental. Support for this field may vary between agent implementations
   193‚Üí   193‚ÜíExample:
   194‚Üí   194‚Üíallowed-tools: Bash(git:*) Bash(jq:*) Read
   195‚Üí   195‚Üí‚Äã
   196‚Üí   196‚ÜíBody content
   197‚Üí   197‚ÜíThe Markdown body after the frontmatter contains the skill instructions. There are no format restrictions. Write whatever helps agents perform the task effectively.
   198‚Üí   198‚ÜíRecommended sections:
   199‚Üí   199‚ÜíStep-by-step instructions
   200‚Üí   200‚ÜíExamples of inputs and outputs
   201‚Üí   201‚ÜíCommon edge cases
   202‚Üí   202‚ÜíNote that the agent will load this entire file once it‚Äôs decided to activate a skill. Consider splitting longer SKILL.md content into referenced files.
   203‚Üí   203‚Üí‚Äã
   204‚Üí   204‚ÜíOptional directories
   205‚Üí   205‚Üí‚Äã
   206‚Üí   206‚Üíscripts/
   207‚Üí   207‚ÜíContains executable code that agents can run. Scripts should:
   208‚Üí   208‚ÜíBe self-contained or clearly document dependencies
   209‚Üí   209‚ÜíInclude helpful error messages
   210‚Üí   210‚ÜíHandle edge cases gracefully
   211‚Üí   211‚ÜíSupported languages depend on the agent implementation. Common options include Python, Bash, and JavaScript.
   212‚Üí   212‚Üí‚Äã
   213‚Üí   213‚Üíreferences/
   214‚Üí   214‚ÜíContains additional documentation that agents can read when needed:
   215‚Üí   215‚ÜíREFERENCE.md - Detailed technical reference
   216‚Üí   216‚ÜíFORMS.md - Form templates or structured data formats
   217‚Üí   217‚ÜíDomain-specific files (finance.md, legal.md, etc.)
   218‚Üí   218‚ÜíKeep individual reference files focused. Agents load these on demand, so smaller files mean less use of context.
   219‚Üí   219‚Üí‚Äã
   220‚Üí   220‚Üíassets/
   221‚Üí   221‚ÜíContains static resources:
   222‚Üí   222‚ÜíTemplates (document templates, configuration templates)
   223‚Üí   223‚ÜíImages (diagrams, examples)
   224‚Üí   224‚ÜíData files (lookup tables, schemas)
   225‚Üí   225‚Üí‚Äã
   226‚Üí   226‚ÜíProgressive disclosure
   227‚Üí   227‚ÜíSkills should be structured for efficient use of context:
   228‚Üí   228‚ÜíMetadata (~100 tokens): The name and description fields are loaded at startup for all skills
   229‚Üí   229‚ÜíInstructions (< 5000 tokens recommended): The full SKILL.md body is loaded when the skill is activated
   230‚Üí   230‚ÜíResources (as needed): Files (e.g. those in scripts/, references/, or assets/) are loaded only when required
   231‚Üí   231‚ÜíKeep your main SKILL.md under 500 lines. Move detailed reference material to separate files.
   232‚Üí   232‚Üí‚Äã
   233‚Üí   233‚ÜíFile references
   234‚Üí   234‚ÜíWhen referencing other files in your skill, use relative paths from the skill root:
   235‚Üí   235‚ÜíSee [the reference guide](references/REFERENCE.md) for details.
   236‚Üí   236‚Üí
   237‚Üí   237‚ÜíRun the extraction script:
   238‚Üí   238‚Üíscripts/extract.py
   239‚Üí   239‚ÜíKeep file references one level deep from SKILL.md. Avoid deeply nested reference chains.
   240‚Üí   240‚Üí‚Äã
   241‚Üí   241‚ÜíValidation
   242‚Üí   242‚ÜíUse the skills-ref reference library to validate your skills:
   243‚Üí   243‚Üískills-ref validate ./my-skill
   244‚Üí   244‚ÜíThis checks that your SKILL.md frontmatter is valid and follows all naming conventions.

   250‚Üí   250‚ÜíHow to add Agent Skills support to your agent or tool.
   251‚Üí   251‚Üí
   252‚Üí   252‚ÜíThis guide explains how to add skills support to an AI agent or development tool.
   253‚Üí   253‚Üí‚Äã
   254‚Üí   254‚ÜíIntegration approaches
   255‚Üí   255‚ÜíThe two main approaches to integrating skills are:
   256‚Üí   256‚ÜíFilesystem-based agents operate within a computer environment (bash/unix) and represent the most capable option. Skills are activated when models issue shell commands like cat /path/to/my-skill/SKILL.md. Bundled resources are accessed through shell commands.
   257‚Üí   257‚ÜíTool-based agents function without a dedicated computer environment. Instead, they implement tools allowing models to trigger skills and access bundled assets. The specific tool implementation is up to the developer.
   258‚Üí   258‚Üí‚Äã
   259‚Üí   259‚ÜíOverview
   260‚Üí   260‚ÜíA skills-compatible agent needs to:
   261‚Üí   261‚ÜíDiscover skills in configured directories
   262‚Üí   262‚ÜíLoad metadata (name and description) at startup
   263‚Üí   263‚ÜíMatch user tasks to relevant skills
   264‚Üí   264‚ÜíActivate skills by loading full instructions
   265‚Üí   265‚ÜíExecute scripts and access resources as needed
   266‚Üí   266‚Üí‚Äã
   267‚Üí   267‚ÜíSkill discovery
   268‚Üí   268‚ÜíSkills are folders containing a SKILL.md file. Your agent should scan configured directories for valid skills.
   269‚Üí   269‚Üí‚Äã
   270‚Üí   270‚ÜíLoading metadata
   271‚Üí   271‚ÜíAt startup, parse only the frontmatter of each SKILL.md file. This keeps initial context usage low.
   272‚Üí   272‚Üí‚Äã
   273‚Üí   273‚ÜíParsing frontmatter
   274‚Üí   274‚Üífunction parseMetadata(skillPath):
   275‚Üí   275‚Üí    content = readFile(skillPath + "/SKILL.md")
   276‚Üí   276‚Üí    frontmatter = extractYAMLFrontmatter(content)
   277‚Üí   277‚Üí
   278‚Üí   278‚Üí    return {
   279‚Üí   279‚Üí        name: frontmatter.name,
   280‚Üí   280‚Üí        description: frontmatter.description,
   281‚Üí   281‚Üí        path: skillPath
   282‚Üí   282‚Üí    }
   283‚Üí   283‚Üí‚Äã
   284‚Üí   284‚ÜíInjecting into context
   285‚Üí   285‚ÜíInclude skill metadata in the system prompt so the model knows what skills are available.
   286‚Üí   286‚ÜíFollow your platform‚Äôs guidance for system prompt updates. For example, for Claude models, the recommended format uses XML:
   287‚Üí   287‚Üí<available_skills>
   288‚Üí   288‚Üí  <skill>
   289‚Üí   289‚Üí    <name>pdf-processing</name>
   290‚Üí   290‚Üí    <description>Extracts text and tables from PDF files, fills forms, merges documents.</description>
   291‚Üí   291‚Üí    <location>/path/to/skills/pdf-processing/SKILL.md</location>
   292‚Üí   292‚Üí  </skill>
   293‚Üí   293‚Üí  <skill>
   294‚Üí   294‚Üí    <name>data-analysis</name>
   295‚Üí   295‚Üí    <description>Analyzes datasets, generates charts, and creates summary reports.</description>
   296‚Üí   296‚Üí    <location>/path/to/skills/data-analysis/SKILL.md</location>
   297‚Üí   297‚Üí  </skill>
   298‚Üí   298‚Üí</available_skills>
   299‚Üí   299‚ÜíFor filesystem-based agents, include the location field with the absolute path to the SKILL.md file. For tool-based agents, the location can be omitted.
   300‚Üí   300‚ÜíKeep metadata concise. Each skill should add roughly 50-100 tokens to the context.
   301‚Üí   301‚Üí‚Äã
   302‚Üí   302‚ÜíSecurity considerations
   303‚Üí   303‚ÜíScript execution introduces security risks. Consider:
   304‚Üí   304‚ÜíSandboxing: Run scripts in isolated environments
   305‚Üí   305‚ÜíAllowlisting: Only execute scripts from trusted skills
   306‚Üí   306‚ÜíConfirmation: Ask users before running potentially dangerous operations
   307‚Üí   307‚ÜíLogging: Record all script executions for auditing
   308‚Üí   308‚Üí‚Äã
   309‚Üí   309‚ÜíReference implementation
   310‚Üí   310‚ÜíThe skills-ref library provides Python utilities and a CLI for working with skills.
   311‚Üí   311‚ÜíFor example:
   312‚Üí   312‚ÜíValidate a skill directory:
   313‚Üí   313‚Üískills-ref validate <path>
   314‚Üí   314‚ÜíGenerate <available_skills> XML for agent prompts:
   315‚Üí   315‚Üískills-ref to-prompt <path>...
   316‚Üí   316‚ÜíUse the library source code as a reference implementation.

   481‚Üí   481‚ÜíWriting effective descriptions
   482‚Üí   482‚ÜíThe description field enables Skill discovery and should include both what the Skill does and when to use it.
   483‚Üí   483‚Üí
   484‚Üí   484‚ÜíAlways write in third person. The description is injected into the system prompt, and inconsistent point-of-view can cause discovery problems.
   485‚Üí   485‚Üí
   486‚Üí   486‚ÜíGood: "Processes Excel files and generates reports"
   487‚Üí   487‚ÜíAvoid: "I can help you process Excel files"
   488‚Üí   488‚ÜíAvoid: "You can use this to process Excel files"
   489‚Üí   489‚ÜíBe specific and include key terms. Include both what the Skill does and specific triggers/contexts for when to use it.
   490‚Üí   490‚Üí
   491‚Üí   491‚ÜíEach Skill has exactly one description field. The description is critical for skill selection: Claude uses it to choose the right Skill from potentially 100+ available Skills.    655‚Üí   655‚ÜíBreak complex operations into clear, sequential steps. For particularly complex workflows, provide a checklist that Claude can copy into its response and check off as it progresses.

   657‚Üí   657‚ÜíExample 1: Research synthesis workflow (for Skills without code):
   658‚Üí   658‚Üí
   659‚Üí   659‚Üí## Research synthesis workflow
   660‚Üí   660‚Üí
   661‚Üí   661‚ÜíCopy this checklist and track your progress:
   662‚Üí   662‚Üí
   663‚Üí   663‚Üí```
   664‚Üí   664‚ÜíResearch Progress:
   665‚Üí   665‚Üí- [ ] Step 1: Read all source documents
   666‚Üí   666‚Üí- [ ] Step 2: Identify key themes
   667‚Üí   667‚Üí- [ ] Step 3: Cross-reference claims
   668‚Üí   668‚Üí- [ ] Step 4: Create structured summary
   669‚Üí   669‚Üí- [ ] Step 5: Verify citations
   670‚Üí   670‚Üí```
   671‚Üí   671‚Üí
   672‚Üí   672‚Üí**Step 1: Read all source documents**
   673‚Üí   673‚Üí
   674‚Üí   674‚ÜíReview each document in the `sources/` directory. Note the main arguments and supporting evidence.
   675‚Üí   675‚Üí
   676‚Üí   676‚Üí**Step 2: Identify key themes**
   677‚Üí   677‚Üí
   678‚Üí   678‚ÜíLook for patterns across sources. What themes appear repeatedly? Where do sources agree or disagree?
   679‚Üí   679‚Üí
   680‚Üí   680‚Üí**Step 3: Cross-reference claims**
   681‚Üí   681‚Üí
   682‚Üí   682‚ÜíFor each major claim, verify it appears in the source material. Note which source supports each point.
   683‚Üí   683‚Üí
   684‚Üí   684‚Üí**Step 4: Create structured summary**
   685‚Üí   685‚Üí
   686‚Üí   686‚ÜíOrganize findings by theme. Include:
   687‚Üí   687‚Üí- Main claim
   688‚Üí   688‚Üí- Supporting evidence from sources
   689‚Üí   689‚Üí- Conflicting viewpoints (if any)
   690‚Üí   690‚Üí
   691‚Üí   691‚Üí**Step 5: Verify citations**
   692‚Üí   692‚Üí
   693‚Üí   693‚ÜíCheck that every claim references the correct source document. If citations are incomplete, return to Step 3.
   694‚Üí   694‚ÜíThis example shows how workflows apply to analysis tasks that don't require code. The checklist pattern works for any complex, multi-step process.

   696‚Üí   696‚ÜíExample 2: PDF form filling workflow (for Skills with code):
   697‚Üí   697‚Üí
   698‚Üí   698‚Üí## PDF form filling workflow
   699‚Üí   699‚Üí
   700‚Üí   700‚ÜíCopy this checklist and check off items as you complete them:
   701‚Üí   701‚Üí
   702‚Üí   702‚Üí```
   703‚Üí   703‚ÜíTask Progress:
   704‚Üí   704‚Üí- [ ] Step 1: Analyze the form (run analyze_form.py)
   705‚Üí   705‚Üí- [ ] Step 2: Create field mapping (edit fields.json)
   706‚Üí   706‚Üí- [ ] Step 3: Validate mapping (run validate_fields.py)
   707‚Üí   707‚Üí- [ ] Step 4: Fill the form (run fill_form.py)
   708‚Üí   708‚Üí- [ ] Step 5: Verify output (run verify_output.py)
   709‚Üí   709‚Üí```
   710‚Üí   710‚Üí
   711‚Üí   711‚Üí**Step 1: Analyze the form**
   712‚Üí   712‚Üí
   713‚Üí   713‚ÜíRun: `python scripts/analyze_form.py input.pdf`
   714‚Üí   714‚Üí
   715‚Üí   715‚ÜíThis extracts form fields and their locations, saving to `fields.json`.
   716‚Üí   716‚Üí
   717‚Üí   717‚Üí**Step 2: Create field mapping**
   718‚Üí   718‚Üí
   719‚Üí   719‚ÜíEdit `fields.json` to add values for each field.
   720‚Üí   720‚Üí
   721‚Üí   721‚Üí**Step 3: Validate mapping**
   722‚Üí   722‚Üí
   723‚Üí   723‚ÜíRun: `python scripts/validate_fields.py fields.json`
   724‚Üí   724‚Üí
   725‚Üí   725‚ÜíFix any validation errors before continuing.
   726‚Üí   726‚Üí
   727‚Üí   727‚Üí**Step 4: Fill the form**
   728‚Üí   728‚Üí
   729‚Üí   729‚ÜíRun: `python scripts/fill_form.py input.pdf fields.json output.pdf`
   730‚Üí   730‚Üí
   731‚Üí   731‚Üí**Step 5: Verify output**
   732‚Üí   732‚Üí
   733‚Üí   733‚ÜíRun: `python scripts/verify_output.py output.pdf`
   734‚Üí   734‚Üí
   735‚Üí   735‚ÜíIf verification fails, return to Step 2.
   736‚Üí   736‚ÜíClear steps prevent Claude from skipping critical validation. The checklist helps both Claude and you track progress through multi-step workflows.

   921‚Üí   921‚ÜíBuild evaluations first

   923‚Üí   923‚Üí
   924‚Üí   924‚ÜíEvaluation-driven development:
   925‚Üí   925‚Üí
   926‚Üí   926‚ÜíIdentify gaps: Run Claude on representative tasks without a Skill. Document specific failures or missing context
   927‚Üí   927‚ÜíCreate evaluations: Build three scenarios that test these gaps
   928‚Üí   928‚ÜíEstablish baseline: Measure Claude's performance without the Skill
   929‚Üí   929‚ÜíWrite minimal instructions: Create just enough content to address the gaps and pass evaluations
   930‚Üí   930‚ÜíIterate: Execute evaluations, compare against baseline, and refine
   931‚Üí   931‚ÜíThis approach ensures you're solving actual problems rather than anticipating requirements that may never materialize.

   947‚Üí   947‚ÜíDevelop Skills iteratively with Claude
   948‚Üí   948‚ÜíThe most effective Skill development process involves Claude itself. Work with one instance of Claude ("Claude A") to create a Skill that will be used by other instances ("Claude B"). Claude A helps you design and refine instructions, while Claude B tests them in real tasks. This works because Claude models understand both how to write effective agent instructions and what information agents need.
   949‚Üí   949‚Üí
   950‚Üí   950‚ÜíCreating a new Skill:
   951‚Üí   951‚Üí
   952‚Üí   952‚ÜíComplete a task without a Skill: Work through a problem with Claude A using normal prompting. As you work, you'll naturally provide context, explain preferences, and share procedural knowledge. Notice what information you repeatedly provide.
   953‚Üí   953‚Üí
   954‚Üí   954‚ÜíIdentify the reusable pattern: After completing the task, identify what context you provided that would be useful for similar future tasks.
   955‚Üí   955‚Üí
   956‚Üí   956‚ÜíExample: If you worked through a BigQuery analysis, you might have provided table names, field definitions, filtering rules (like "always exclude test accounts"), and common query patterns.
   957‚Üí   957‚Üí
   958‚Üí   958‚ÜíAsk Claude A to create a Skill: "Create a Skill that captures this BigQuery analysis pattern we just used. Include the table schemas, naming conventions, and the rule about filtering test accounts."
   959‚Üí   959‚Üí
   960‚Üí   960‚ÜíClaude models understand the Skill format and structure natively. You don't need special system prompts or a "writing skills" skill to get Claude to help create Skills. Simply ask Claude to create a Skill and it will generate properly structured SKILL.md content with appropriate frontmatter and body content.
   961‚Üí   961‚Üí
   962‚Üí   962‚ÜíReview for conciseness: Check that Claude A hasn't added unnecessary explanations. Ask: "Remove the explanation about what win rate means - Claude already knows that."
   963‚Üí   963‚Üí
   964‚Üí   964‚ÜíImprove information architecture: Ask Claude A to organize the content more effectively. For example: "Organize this so the table schema is in a separate reference file. We might add more tables later."
   965‚Üí   965‚Üí
   966‚Üí   966‚ÜíTest on similar tasks: Use the Skill with Claude B (a fresh instance with the Skill loaded) on related use cases. Observe whether Claude B finds the right information, applies rules correctly, and handles the task successfully.
   967‚Üí   967‚Üí
   968‚Üí   968‚ÜíIterate based on observation: If Claude B struggles or misses something, return to Claude A with specifics: "When Claude used this Skill, it forgot to filter by date for Q4. Should we add a section about date filtering patterns?"

   970‚Üí   970‚ÜíIterating on existing Skills:
   971‚Üí   971‚Üí

   973‚Üí   973‚Üí
   974‚Üí   974‚ÜíWorking with Claude A (the expert who helps refine the Skill)
   975‚Üí   975‚ÜíTesting with Claude B (the agent using the Skill to perform real work)
   976‚Üí   976‚ÜíObserving Claude B's behavior and bringing insights back to Claude A
   977‚Üí   977‚ÜíUse the Skill in real workflows: Give Claude B (with the Skill loaded) actual tasks, not test scenarios
   978‚Üí   978‚Üí
   979‚Üí   979‚ÜíObserve Claude B's behavior: Note where it struggles, succeeds, or makes unexpected choices
   980‚Üí   980‚Üí
   981‚Üí   981‚ÜíExample observation: "When I asked Claude B for a regional sales report, it wrote the query but forgot to filter out test accounts, even though the Skill mentions this rule."
   982‚Üí   982‚Üí
   983‚Üí   983‚ÜíReturn to Claude A for improvements: Share the current SKILL.md and describe what you observed. Ask: "I noticed Claude B forgot to filter test accounts when I asked for a regional report. The Skill mentions filtering, but maybe it's not prominent enough?"
   984‚Üí   984‚Üí
   985‚Üí   985‚ÜíReview Claude A's suggestions: Claude A might suggest reorganizing to make rules more prominent, using stronger language like "MUST filter" instead of "always filter", or restructuring the workflow section.
   986‚Üí   986‚Üí
   987‚Üí   987‚ÜíApply and test changes: Update the Skill with Claude A's refinements, then test again with Claude B on similar requests
   988‚Üí   988‚Üí
   989‚Üí   989‚ÜíRepeat based on usage: Continue this observe-refine-test cycle as you encounter new scenarios. Each iteration improves the Skill based on real agent behavior, not assumptions.
   990‚Üí   990‚Üí
   991‚Üí   991‚ÜíGathering team feedback:
   992‚Üí   992‚Üí
   993‚Üí   993‚ÜíShare Skills with teammates and observe their usage
   994‚Üí   994‚ÜíAsk: Does the Skill activate when expected? Are instructions clear? What's missing?
   995‚Üí   995‚ÜíIncorporate feedback to address blind spots in your own usage patterns
   996‚Üí   996‚ÜíWhy this approach works: Claude A understands agent needs, you provide domain expertise, Claude B reveals gaps through real usage, and iterative refinement improves Skills based on observed behavior rather than assumptions.
   997‚Üí   997‚Üí
   998‚Üí   998‚ÜíObserve how Claude navigates Skills
   999‚Üí   999‚ÜíAs you iterate on Skills, pay attention to how Claude actually uses them in practice. Watch for:
  1000‚Üí  1000‚Üí
  1001‚Üí  1001‚ÜíUnexpected exploration paths: Does Claude read files in an order you didn't anticipate? This might indicate your structure isn't as intuitive as you thought
  1002‚Üí  1002‚ÜíMissed connections: Does Claude fail to follow references to important files? Your links might need to be more explicit or prominent
  1003‚Üí  1003‚ÜíOverreliance on certain sections: If Claude repeatedly reads the same file, consider whether that content should be in the main SKILL.md instead
  1004‚Üí  1004‚ÜíIgnored content: If Claude never accesses a bundled file, it might be unnecessary or poorly signaled in the main instructions
  1005‚Üí  1005‚ÜíIterate based on these observations rather than assumptions. The 'name' and 'description' in your Skill's metadata are particularly critical. Claude uses these when deciding whether to trigger the Skill in response to the current task. Make sure they clearly describe what the Skill does and when it should be used.

  1007‚Üí  1007‚ÜíAnti-patterns to avoid
  1008‚Üí  1008‚ÜíAvoid Windows-style paths
  1009‚Üí  1009‚ÜíAlways use forward slashes in file paths, even on Windows:
  1010‚Üí  1010‚Üí
  1011‚Üí  1011‚Üí‚úì Good: scripts/helper.py, reference/guide.md
  1012‚Üí  1012‚Üí‚úó Avoid: scripts\helper.py, reference\guide.md
  1013‚Üí  1013‚ÜíUnix-style paths work across all platforms, while Windows-style paths cause errors on Unix systems.
  1014‚Üí  1014‚Üí
  1015‚Üí  1015‚ÜíAvoid offering too many options
  1016‚Üí  1016‚ÜíDon't present multiple approaches unless necessary:
  1017‚Üí  1017‚Üí
  1018‚Üí  1018‚Üí**Bad example: Too many choices** (confusing):
  1019‚Üí  1019‚Üí"You can use pypdf, or pdfplumber, or PyMuPDF, or pdf2image, or..."
  1020‚Üí  1020‚Üí
  1021‚Üí  1021‚Üí**Good example: Provide a default** (with escape hatch):
  1022‚Üí  1022‚Üí"Use pdfplumber for text extraction:
  1023‚Üí  1023‚Üí```python
  1024‚Üí  1024‚Üíimport pdfplumber
  1025‚Üí  1025‚Üí```
  1026‚Üí  1026‚Üí
  1027‚Üí  1027‚ÜíFor scanned PDFs requiring OCR, use pdf2image with pytesseract instead."
  1028‚Üí  1028‚ÜíAdvanced: Skills with executable code

  1030‚Üí  1030‚Üí
  1031‚Üí  1031‚ÜíSolve, don't punt
  1032‚Üí  1032‚ÜíWhen writing scripts for Skills, handle error conditions rather than punting to Claude.
  1033‚Üí  1033‚Üí
  1034‚Üí  1034‚ÜíGood example: Handle errors explicitly:
  1035‚Üí  1035‚Üí
  1036‚Üí  1036‚Üídef process_file(path):
  1037‚Üí  1037‚Üí    """Process a file, creating it if it doesn't exist."""
  1038‚Üí  1038‚Üí    try:
  1039‚Üí  1039‚Üí        with open(path) as f:
  1040‚Üí  1040‚Üí            return f.read()
  1041‚Üí  1041‚Üí    except FileNotFoundError:
  1042‚Üí  1042‚Üí        # Create file with default content instead of failing
  1043‚Üí  1043‚Üí        print(f"File {path} not found, creating default")
  1044‚Üí  1044‚Üí        with open(path, 'w') as f:
  1045‚Üí  1045‚Üí            f.write('')
  1046‚Üí  1046‚Üí        return ''
  1047‚Üí  1047‚Üí    except PermissionError:
  1048‚Üí  1048‚Üí        # Provide alternative instead of failing
  1049‚Üí  1049‚Üí        print(f"Cannot access {path}, using default")
  1050‚Üí  1050‚Üí        return ''
  1051‚Üí  1051‚ÜíBad example: Punt to Claude:
  1052‚Üí  1052‚Üí
  1053‚Üí  1053‚Üídef process_file(path):
  1054‚Üí  1054‚Üí    # Just fail and let Claude figure it out
  1055‚Üí  1055‚Üí    return open(path).read()
  1056‚Üí  1056‚ÜíConfiguration parameters should also be justified and documented to avoid "voodoo constants" (Ousterhout's law). If you don't know the right value, how will Claude determine it?
  1057‚Üí  1057‚Üí
  1058‚Üí  1058‚ÜíGood example: Self-documenting:
  1059‚Üí  1059‚Üí
  1060‚Üí  1060‚Üí# HTTP requests typically complete within 30 seconds
  1061‚Üí  1061‚Üí# Longer timeout accounts for slow connections
  1062‚Üí  1062‚ÜíREQUEST_TIMEOUT = 30
  1063‚Üí  1063‚Üí
  1064‚Üí  1064‚Üí# Three retries balances reliability vs speed
  1065‚Üí  1065‚Üí# Most intermittent failures resolve by the second retry
  1066‚Üí  1066‚ÜíMAX_RETRIES = 3
  1067‚Üí  1067‚ÜíBad example: Magic numbers:
  1068‚Üí  1068‚Üí
  1069‚Üí  1069‚ÜíTIMEOUT = 47  # Why 47?
  1070‚Üí  1070‚ÜíRETRIES = 5   # Why 5?

  1075‚Üí  1075‚Üí

  1136‚Üí  1136‚ÜíCreate verifiable intermediate outputs
  1137‚Üí  1137‚ÜíWhen Claude performs complex, open-ended tasks, it can make mistakes. The "plan-validate-execute" pattern catches errors early by having Claude first create a plan in a structured format, then validate that plan with a script before executing it.
  1138‚Üí  1138‚Üí
  1139‚Üí  1139‚ÜíExample: Imagine asking Claude to update 50 form fields in a PDF based on a spreadsheet. Without validation, Claude might reference non-existent fields, create conflicting values, miss required fields, or apply updates incorrectly.
  1140‚Üí  1140‚Üí
  1141‚Üí  1141‚ÜíSolution: Use the workflow pattern shown above (PDF form filling), but add an intermediate changes.json file that gets validated before applying changes. The workflow becomes: analyze ‚Üí create plan file ‚Üí validate plan ‚Üí execute ‚Üí verify.
  1142‚Üí  1142‚Üí
  1143‚Üí  1143‚ÜíWhy this pattern works:
  1144‚Üí  1144‚Üí
  1145‚Üí  1145‚ÜíCatches errors early: Validation finds problems before changes are applied
  1146‚Üí  1146‚ÜíMachine-verifiable: Scripts provide objective verification
  1147‚Üí  1147‚ÜíReversible planning: Claude can iterate on the plan without touching originals
  1148‚Üí  1148‚ÜíClear debugging: Error messages point to specific problems
  1149‚Üí  1149‚ÜíWhen to use: Batch operations, destructive changes, complex validation rules, high-stakes operations.
  1150‚Üí  1150‚Üí
  1151‚Üí  1151‚ÜíImplementation tip: Make validation scripts verbose with specific error messages like "Field 'signature_date' not found. Available fields: customer_name, order_total, signature_date_signed" to help Claude fix issues.

  1164‚Üí  1164‚Üí
  1165‚Üí  1165‚ÜíHow Claude accesses Skills:
  1166‚Üí  1166‚Üí

  1170‚Üí  1170‚ÜíNo context penalty for large files: Reference files, data, or documentation don't consume context tokens until actually read
  1171‚Üí  1171‚ÜíFile paths matter: Claude navigates your skill directory like a filesystem. Use forward slashes (reference/guide.md), not backslashes
  1172‚Üí  1172‚ÜíName files descriptively: Use names that indicate content: form_validation_rules.md, not doc2.md
  1173‚Üí  1173‚ÜíOrganize for discovery: Structure directories by domain or feature
  1174‚Üí  1174‚ÜíGood: reference/finance.md, reference/sales.md
  1175‚Üí  1175‚ÜíBad: docs/file1.md, docs/file2.md
  1176‚Üí  1176‚ÜíBundle comprehensive resources: Include complete API docs, extensive examples, large datasets; no context penalty until accessed
  1177‚Üí  1177‚ÜíPrefer scripts for deterministic operations: Write validate_form.py rather than asking Claude to generate validation code
  1178‚Üí  1178‚ÜíMake execution intent clear:
  1179‚Üí  1179‚Üí"Run analyze_form.py to extract fields" (execute)
  1180‚Üí  1180‚Üí"See analyze_form.py for the extraction algorithm" (read as reference)
