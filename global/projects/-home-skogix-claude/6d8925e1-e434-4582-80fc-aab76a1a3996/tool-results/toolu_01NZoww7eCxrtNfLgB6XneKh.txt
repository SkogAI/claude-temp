131-   131→## Configuration
132-   132→
133-   133→Hooks are defined in JSON settings files. The configuration has three levels of nesting:
134-   134→
135:   135→1. Choose a [hook event](#hook-events) to respond to, like `PreToolUse` or `Stop`
136-   136→2. Add a [matcher group](#matcher-patterns) to filter when it fires, like "only for the Bash tool"
137-   137→3. Define one or more [hook handlers](#hook-handler-fields) to run when matched
138-   138→
139-   139→See [How a hook resolves](#how-a-hook-resolves) above for a complete walkthrough with an annotated example.
140-   140→
--
146-   146→
147-   147→Where you define a hook determines its scope:
148-   148→
149-   149→| Location                                                   | Scope                         | Shareable                          |
150-   150→| :--------------------------------------------------------- | :---------------------------- | :--------------------------------- |
151:   151→| `~/.claude/settings.json`                                  | All your projects             | No, local to your machine          |
152:   152→| `.claude/settings.json`                                    | Single project                | Yes, can be committed to the repo  |
153-   153→| `.claude/settings.local.json`                              | Single project                | No, gitignored                     |
154-   154→| Managed policy settings                                    | Organization-wide             | Yes, admin-controlled              |
155-   155→| [Plugin](/en/plugins) `hooks/hooks.json`                   | When plugin is enabled        | Yes, bundled with the plugin       |
156-   156→| [Skill](/en/skills) or [agent](/en/sub-agents) frontmatter | While the component is active | Yes, defined in the component file |
157-   157→
--
161-   161→
162-   162→The `matcher` field is a regex string that filters when hooks fire. Use `"*"`, `""`, or omit `matcher` entirely to match all occurrences. Each event type matches on a different field:
163-   163→
164-   164→| Event                                                                                           | What the matcher filters  | Example matcher values                                                             |
165-   165→| :---------------------------------------------------------------------------------------------- | :------------------------ | :--------------------------------------------------------------------------------- |
166:   166→| `PreToolUse`, `PostToolUse`, `PostToolUseFailure`, `PermissionRequest`                          | tool name                 | `Bash`, `Edit\|Write`, `mcp__.*`                                                   |
167-   167→| `SessionStart`                                                                                  | how the session started   | `startup`, `resume`, `clear`, `compact`                                            |
168-   168→| `SessionEnd`                                                                                    | why the session ended     | `clear`, `logout`, `prompt_input_exit`, `bypass_permissions_disabled`, `other`     |
169-   169→| `Notification`                                                                                  | notification type         | `permission_prompt`, `idle_prompt`, `auth_success`, `elicitation_dialog`           |
170-   170→| `SubagentStart`                                                                                 | agent type                | `Bash`, `Explore`, `Plan`, or custom agent names                                   |
171-   171→| `PreCompact`                                                                                    | what triggered compaction | `manual`, `auto`                                                                   |

197-   197→
198-   198→`UserPromptSubmit`, `Stop`, `TeammateIdle`, `TaskCompleted`, `WorktreeCreate`, and `WorktreeRemove` don't support matchers and always fire on every occurrence. If you add a `matcher` field to these events, it is silently ignored.

423-   423→
424-   424→The exit code from your hook command tells Claude Code whether the action should proceed, be blocked, or be ignored.
425-   425→
426-   426→**Exit 0** means success. Claude Code parses stdout for [JSON output fields](#json-output). JSON output is only processed on exit 0. For most events, stdout is only shown in verbose mode (`Ctrl+O`). The exceptions are `UserPromptSubmit` and `SessionStart`, where stdout is added as context that Claude can see and act on.
427-   427→
428:   428→**Exit 2** means a blocking error. Claude Code ignores stdout and any JSON in it. Instead, stderr text is fed back to Claude as an error message. The effect depends on the event: `PreToolUse` blocks the tool call, `UserPromptSubmit` rejects the prompt, and so on. See [exit code 2 behavior](#exit-code-2-behavior-per-event) for the full list.
429-   429→
430-   430→**Any other exit code** is a non-blocking error. stderr is shown in verbose mode (`Ctrl+O`) and execution continues.

816:   816→#### PreToolUse decision control
817-   817→
818:   818→`PreToolUse` hooks can control whether a tool call proceeds. Unlike other hooks that use a top-level `decision` field, PreToolUse returns its decision inside a `hookSpecificOutput` object. 819-   819→
820-   820→| Field                      | Description                                                                                                                                      |
821-   821→| :------------------------- | :----------------------------------------------------------------------------------------------------------------------------------------------- |
822-   822→| `permissionDecision`       | `"allow"` bypasses the permission system, `"deny"` prevents the tool call, `"ask"` prompts the user to confirm                                   |
823-   823→| `permissionDecisionReason` | For `"allow"` and `"ask"`, shown to the user but not Claude. For `"deny"`, shown to Claude                                                       |
--
825-   825→| `additionalContext`        | String added to Claude's context before the tool executes                                                                                        |

827-   827→```json  theme={null}
828-   828→{
829-   829→  "hookSpecificOutput": {
830:   830→    "hookEventName": "PreToolUse",
831-   831→    "permissionDecision": "allow",
832-   832→    "permissionDecisionReason": "My reason here",
833-   833→    "updatedInput": {

835-   835→    },

837-   837→  }
838-   838→}
839-   839→```

1703-  1703→Async hooks have several constraints compared to synchronous hooks:
1704-  1704→
1705-  1705→* Only `type: "command"` hooks support `async`. Prompt-based hooks cannot run asynchronously.
